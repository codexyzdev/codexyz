### ¿Cómo manejaremos los cambios en el futuro sin perder datos?

El error que nos obligó a resetear hoy fue haber **modificado un archivo de migración que ya había sido aplicado** (`migration.sql`). Prisma detectó que el archivo cambió después de ejecutarse y, por seguridad, pidió un reset.

**El flujo correcto para evitar esto es:**

1.  **Nunca tocar migraciones pasadas**: Si una migración ya se aplicó (incluso si tuvo un error), no se edita el archivo `.sql` existente.
2.  **Crear una NUEVA migración para corregir**: En lugar de editar el pasado, creamos una nueva migración (`prisma migrate dev --name fix_issue`) que aplique los cambios o correcciones necesarios sobre el estado actual.
3.  **Migraciones Aditivas**: Siempre agregamos cambios hacia adelante.

**Ejemplo práctico:**
Si hubiéramos seguido este flujo hoy, en lugar de editar el archivo `2026...add_rls_policies` para arreglar el tipo de dato UUID, hubiéramos creado una nueva migración `2026...fix_rls_types` con el comando `ALTER POLICY...`.

De esta forma, la base de datos avanza paso a paso sin necesidad de borrarse nunca. Queda registrado como protocolo estricto.
